Overview

The Job Registry is responsible for managing all jobs (fep3::arya::IJobRegistry::IJob) of a FEP Participant and for making them available to other components like the Scheduler Service. Using the Job Configuration registered jobs can also be reconfigured without recompilation.

Component Interface

The fep3::arya::IJobRegistry interface offers the following functionality:

  • Adding and removing Jobs to/from the Job Registry.

  • Retrieve information about all currently registered Jobs.

  • Retrieve the Jobs itself. The retrieved Jobs can be used for scheduling.

Note

A job has to be added/removed to/from the Job Registry when the Implementing the fep3::core::arya::ElementBase::load() and fep3::core::arya::ElementBase::initialize() is:

  • loading (fep3::arya::IElement::loadElement()),

  • unloading (fep3::arya::IElement::unloadElement()),

Job

A fep3::arya::IJobRegistry::IJob is a cyclic processing unit which can be added to the Job Registry. Every Job has a unique name within the Job Registry it is added to.

A Job typically will read signal data from fep3::arya::IDataRegistry::IDataReader, process received data samples and write signal samples to fep3::arya::IDataRegistry::IDataWriter. The configured Active scheduler will trigger all jobs which are registered at the Job Registry according to its scheduling mechanism.

The execution of a Job is separated by the following steps:

  • fep3::arya::IJobRegistry::IJob::executeDataIn() to read sample data from readers.

  • fep3::arya::IJobRegistry::IJob::execute() to do the actual processing.

  • fep3::arya::IJobRegistry::IJob::executeDataOut() to write sample data to writers.

A Job is configured by a fep3::arya::JobConfiguration when adding it to the fep3::arya::IJobRegistry. The main configuration of a Job is the fep3::arya::JobConfiguration::_cycle_sim_time which determines in which intervals a Job is executed by the Active scheduler. For the other configuration options see fep3::arya::JobConfiguration.

Note

The configuration parameter fep3::arya::JobConfiguration::_delay_sim_time is currently not evaluated.

Sample validity for a simulation step

If samples are read from a fep3::arya::IDataRegistry::IDataReader, only samples with a timestamp smaller than the current simulation time, will be provided. Only those samples are considered to be valid for the current simulation step. Samples with a simulation time equal to the current time are accessible from the next simulation step on. With this simulation model there is no data dependency between jobs within one simulation step and all jobs of one simulation step can be calculated in parallel.

Configuration

The Job Registry can be configured using the following properties.

Properties

Name

Code Macro

Default Value

Jobs

FEP3_JOB_REGISTRY_JOBS

No value. Retrieves a job entry subproperty containing the job configuration values as subproperties on job registration.

Note

Job entry sub properties of FEP3_JOB_REGISTRY_JOBS have to be set after the Implementing the fep3::core::arya::ElementBase::load() and fep3::core::arya::ElementBase::initialize() has been loaded (fep3::arya::IElement::loading()) when jobs are registered and before the fep3::arya::ISchedulerService is tensing (fep3::arya::IComponent::tense()) when job nodes and their configurations are parsed and applied.

Job Configuration

During registration a job is configured using a fep3::arya::JobConfiguration. To adapt the behavior of participants without having to recompile, registered jobs may be reconfigured using the FEP3_JOB_REGISTRY_JOBS property node which contains information regarding all registered jobs and their configuration. A job entry equals a property node containing the job name as value and following sub properties containing information regarding the job configuration:

Attribute

Valid values

Description

FEP3_JOB_CYCLE_SIM_TIME_PROPERTY

An integer > 0

The time after which a job is triggered cyclically by the scheduler in microseconds.

FEP3_JOB_DELAY_SIM_TIME_PROPERTY

An integer >= 0

Not yet applied.

FEP3_JOB_MAX_RUNTIME_REAL_TIME_PROPERTY

An integer >= 0

The maximum real time value a job may take to run in microseconds. If a job exceeds the configured value, a configured strategy is executed. In case of value ‘0’, runtime violation strategies are not applied.

FEP3_JOB_RUNTIME_VIOLATION_STRATEGY_PROPERTY

fep3::arya::JobConfiguration::TimeViolationStrategy::ignore_runtime_violation, fep3::arya::JobConfiguration::TimeViolationStrategy::warn_about_runtime_violation, fep3::arya::JobConfiguration::TimeViolationStrategy::skip_output_publish,

The strategy to react to job runtime violations.

Native Implementations

The FEP Participant Library provides a native implementation of a Job Registry, which will be used automatically.
