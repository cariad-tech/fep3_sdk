FEP Component SystemÔÉÅ

The FEP Component System makes a FEP Participant work with a certain set of FEP Components and FEP Component Implementations. The purpose of a FEP Participant is defined by the set of FEP Components it provides, while the behavior of a FEP Participant depends on the FEP Component Implementations that are loaded by the FEP Component System.

For example

  • the purpose of a FEP Participant in the context of FEP is to be a participant of a simulation and integration platform for the development and for the test of highly cross-linked vehicle functions

  • the behavior of a FEP Participant can be e. g.

    • simulation data is transferred via RTI DDS

    • or simulation data is transferred via ZeroMQ.

Using the FEP Component System the purpose and/or the behavior of the FEP Participant can be changed or extended by configuring the FEP Participant to use other FEP Components than the FEP Default Components.

Default FEP Components vs. Native FEP Components vs. Foreign FEP ComponentsÔÉÅ

Native FEP Components are shipped with the FEP Participant Library. The FEP Participant loads a subset of those Native FEP Components by default, i. e. if no FEP Component Configuration File is provided. The FEP Components in such subset are called FEP Default Components. Additionally to the FEP Components in the set of FEP Default Components the FEP Participant Library ships the following Native FEP Components:

FEP Component Implementation

FEP Component

FEP Component Interface

HTTP Service Bus

Service Bus

fep3::arya::IServiceBus

RTI Connext DDS Simulation Bus

Simulation Bus

fep3::arya::ISimulationBus

Besides those Native FEP Components the FEP Participant can be configured to load so-called ‚ÄúForeign FEP Components‚Äù. Such FEP Components may be delivered by other departments of the FEP project or by third parties.

Using the FEP Component SystemÔÉÅ

To make a FEP Participant use other FEP Components than the FEP Default Components, a FEP Component Configuration File has to be provided. This file contains information on which FEP Components shall be used by a FEP Participant.

FEP Component Configuration FileÔÉÅ

The FEP Component Configuration File is an XML file with the following content (exemplary):

<?xml version="1.0" encoding="utf-8"?>
<components xmlns="http://fep.vwgroup.com/fep_sdk/3.0/components">
    <schema_version>1.0.0</schema_version>
    <component>
        <source type="built-in"/>
        <iid>clock_service.arya.fep3.iid</iid>
    </component>
    <component>
        <source type="cpp-plugin">
            my_component_plugin
        </source>
        <iid>service_bus.arya.fep3.iid</iid>
    </component>
    <component>
        <source type="c-plugin">
            third_party_component_plugin
        </source>
        <iid>simulation_bus.arya.fep3.iid</iid>
    </component>
</components>

where

  • the components/component element can occur any number of times

  • the components/component/iid element contains the interface identifier of the component to be loaded

  • the components/component/source element contains information on where to load the component implementation from:

    • the components/component/source@type attribute defines the Component instantiation and takes the values built-in, cpp-plugin or c-plugin

    • if the components/component/source@type attribute is cpp-plugin or c-plugin, the data of components/component/source has to contain the file path of the FEP Component Plugin‚Äôs shared library that provides the component implementation; such file path has to be either absolute or relative to the directory the FEP Component Configuration File resides in

During startup the FEP Component System of the FEP Participant tries to find a FEP Component Configuration File as follows:

  • If the environment variable ‚ÄúFEP3_PARTICIPANT_COMPONENTS_FILE_PATH‚Äù is set and it contains an absolute file path it tries to find a FEP Component Configuration File at this absolute file path.

  • If the environment variable ‚ÄúFEP3_PARTICIPANT_COMPONENTS_FILE_PATH‚Äù is set and it contains a relative file path it tries to find a FEP Component Configuration File at this file path relative to

    • the current working directory

    • the directory where the fep3_participant shared library file resides in

  • If the environment variable ‚ÄúFEP3_PARTICIPANT_COMPONENTS_FILE_PATH‚Äù is not set it tries to find a FEP Component Configuration File with the name ‚Äúfep3_participant.fep_components‚Äù in

    • the current working directory

    • the directory where the fep3_participant shared library file resides in

If a FEP Component Configuration File is found according to those rules, the FEP Components are loaded according to the rules in that file. Otherwise, i. e. if no FEP Component Configuration File is found, the FEP Default Components are loaded.

To ensure consistency, all FEP Participants of a FEP System should use the same FEP Component Configuration File. Therefore the FEP Component Configuration File

  • has to be provided by the FEP System Developer

  • has to be deployed to the FEP Participants by the deployment process of the FEP System.

Component instantiationÔÉÅ

Upon creation of the participant (see Creation), the FEP Components are instantiated by the FEP Component System. The order of the FEP Component instantiation follows the order of the components/component elements in the FEP Component Configuration File.

The FEP Participant Library provides three mechanisms for the instantiation of FEP Components:

  • ‚Äúbuilt-in‚Äù: the FEP Component instance is being created from the pool of FEP Component Implementations that are built into the FEP Participant Library

  • ‚ÄúCPP plugin‚Äù: the FEP Component instance is being created from a Component CPP Plugin

  • ‚ÄúC plugin‚Äù: the FEP Component instance is being created from a Component C Plugin

See Native FEP Components for a list and description of FEP Components that are delivered with the FEP Participant Library.

The destruction of the FEP Components takes place when the participant instance is destroyed. The order of the FEP Component destruction is done in the revers order of the components/component elements in the FEP Component Configuration File.

Calls to the fep3::arya::IComponent interfaceÔÉÅ

When the participant is executed (see Execution), the method fep3::arya::IComponent::createComponent() is called; when the participant execution has finished (State Machine state Final) the method fep3::arya::IComponent::destroyComponent() is called.

All other calls to the methods of the FEP Component‚Äôs interface fep3::arya::IComponent (e. g. fep3::arya::IComponent::initialize()) occur from within the context of the transitions of the FEP Participant‚Äôs State Machine. Component CPP PluginsÔÉÅ

CPP plugins use the C++ interface of the FEP Component System. They offer the highest flexibility as they are not restricted to a subset of FEP Components, i. e. the functionality of a FEP Participant can be changed or extended by any functionality via CPP plugins, even by defining completely new FEP Components. For CPP plugins the FEP Participant Library does not guarantee full backward compatibility, so CPP plugins might have to be rebuilt when switching to a newer version of the FEP Participant Library.

Component C PluginÔÉÅ

C plugins use the C interface of the FEP Component System. They ensure full binary compatibility for mixing debug/release versions and different compilers and compiler settings. The FEP Participant Library guarantees full backward compatibility for C plugins, i. e. future versions of the FEP Participant can use existing C plugins. Therefore FEP recommends using C plugins for connecting FEP Participants to third-party middleware.

Note

Components in C plugins are written in C++ programming language against the C++ FEP Component Interface. Only the hidden interface between the plugin and the FEP Participant Library is implemented in C language.

Creating a FEP Component Plugin in order to provide Foreign FEP ComponentsÔÉÅ

FEP Component Plugins must be provided as shared libraries.

Note

One FEP Component Plugin may contain multiple FEP Components but only one FEP Component Implementation per FEP Component Interface.

FEP Component CPP APIÔÉÅ

To create a Component CPP Plugin, include

#include <fep3/plugin/cpp/cpp_plugin_impl_arya.hpp>

in your plugin‚Äôs C++ source file and implement the functions fep3_plugin_getPluginVersion and fep3_plugin_cpp_arya_getFactory, e. g. like so:

void fep3_plugin_getPluginVersion(void(*callback)(void*, const char*), void* destination)
{
    callback(destination, "my component plugin, version 1.0");
}

fep3::plugin::cpp::ICPPPluginComponentFactory* fep3_plugin_cpp_arya_getFactory()
{
    return new fep3::plugin::cpp::CPPPluginComponentFactory<MyComponentInterface>;
}

To create your FEP Component Implementation include

#include <fep3/components/base/component.h>

and derive your FEP Component Implementation class publicly from

fep3::base::Component<MyComponentInterface>

The FEP Component Interface (‚ÄúMyComponentInterface‚Äù in the examples above) may be one of the interfaces defined by the FEP Participant Library (see FEP Default Components) or any other class calling the macro FEP_COMPONENT_IID, e. g.

class MyComponentInterface
{
protected:
    virtual ~MyComponentInterface() = default;

public:
    FEP_COMPONENT_IID("my_component");

    // methods of the component
    virtual int get() const = 0;
};

Note

The destructor of such FEP Component Interface class shall be protected to prevent destruction by the FEP Participant Developer or the FEP Component Developer.

FEP Component C APIÔÉÅ

To create a Component C Plugin, include

#include <fep3/plugin/c/c_plugin.h>

in your plugin‚Äôs C++ source file and implement the function fep3_plugin_getPluginVersion, e. g. like so:

void fep3_plugin_getPluginVersion(void(*callback)(void*, const char*), void* destination)
{
    callback(destination, "my component plugin, version 1.0");
}

To create your FEP Component Implementation include

#include <fep3/components/base/c_access_wrapper/transferable_component_base.h>

and derive your FEP Component Implementation class publicly from

fep3::plugin::c::TransferableComponentBase<...>

, passing the appropriate FEP Component Interface (e. g. fep3::arya::ISimulationBus) as template parameter.

In your FEP Component Implementation file include the appropriate access wrapper header, e. g. for a Simulation Bus Component

#include <fep3/components/simulation_bus/c_access_wrapper/simulation_bus_c_access_wrapper.h>

and implement the component creation function, e. g. fep3_plugin_c_arya_createSimulationBus, like so:

fep3_plugin_c_CInterfaceError fep3_plugin_c_arya_createSimulationBus
    (fep3_arya_SISimulationBus* access
    , fep3_plugin_c_arya_SISharedBinary shared_binary_access
    , const char* iid
    )
{
    return fep3::plugin::c::wrapper::arya::createSimulationBus
        ([]()
            {
                return new MySimulationBus;
            }
        , access
        , shared_binary_access
        , iid
        );
}

where MySimulationBus is your FEP Component Implementation class.
