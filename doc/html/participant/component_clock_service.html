OverviewÔÉÅ

The Clock Service is responsible for managing one or more different clocks which may represent different times with a different time base. Every registered clock is identified by a unique name. There is one so called Main Clock, which provides the time of the Clock Service. As described in the section Clock Type , clocks are distinguished by a type.

The time of the Clock Service represents the simulation time of the participant.

Component InterfaceÔÉÅ

The fep3::arya::IClockService offers the following functionality:

  • Register and unregister clocks and retrieve information about all registered clocks including the Main Clock.

  • Register and unregister event sinks (fep3::arya::IClock::IEventSink).

  • Get the current time or the type of the Clock Service or of any other registered clock.

Event SinkÔÉÅ

An Event Sink allows synchronous reactions on time reset and time update events. An Event Sink is created by implementing the abstract methods of (fep3::arya::IClock::IEventSink). It can be registered using fep3::arya::IClockService::registerEventSink(). After registration the Event Sink will be notified whenever the Main Clock emits an event.

RPC Service InterfacesÔÉÅ

RPC Clock ServiceÔÉÅ

The fep3::rpc::arya::IRPCClockServiceDef offers the following functionality:

  • Retrieve information about all registered clocks.

  • Get the current time of the Clock Service or of any other registered clock.

RPC Clock Sync MasterÔÉÅ

The Clock Service provides a RPC Clock Sync Master which is the counterpart of the Clock Sync Service and enables the synchronization of a timing master with timing slaves. The RPC Clock Sync Master offers the possibility to register as a synchronization slave as well as to retrieve the current time of the master and its type (fep3::rpc::arya::IRPCClockSyncMasterDef for detail). Also, it can actively synchronize timing slaves. The communication is done via RPC. To establish a clock synchronization the Clock Sync Service has to be used on the slave side. If discrete synchronization is used (e.g. slave_master_on_demand_discrete), all timing slaves are synchronized concurrently. The master will wait until all slaves are successfully synchronized or a timeout occurred. The timeout is configured via FEP3_CLOCK_SERVICE_TIME_UPDATE_TIMEOUT. If the timeout is exceeded the relevant timing slave will be removed from synchronization and an incident is risen but simulation will be continued.

Main ClockÔÉÅ

The Main Clock is the clock providing the current time for a FEP Participant. Calling fep3::arya::IClockService::getTime() will return the time of that clock.

If the current Main Clock is being unregistered, the Clock Service emits a warning level log and sets local_system_realtime as Main Clock.

Clock TypeÔÉÅ

Clocks can either be of type Discrete or Continuous. In the API the Enum fep3::arya::ClockType is used to distinguish the types of clocks.

DiscreteÔÉÅ

A discrete clock actively sets a timestamp and synchronously propagates an internal updating event to all registered Event Sink s. If other slave participants use the Clock Sync Service the time update event will be propagated via RPC. See local_system_simtime for an implementation.

ContinuousÔÉÅ

A continuous clock will provide real time (or simulated real time) on each fep3::arya::IClockService::getTime() call. See local_system_realtime for an implementation.

ConfigurationÔÉÅ

The Clock Service can be configured using the following property. Depending on the specific clock configured as Main Clock, there might be additional configuration options. For details about the native clocks, see the documentation of the Native Implementations.

If the participant you are configuring is a timing slave you have to set the Main Clock to one of the slave clocks, described in the section of the Clock Sync Service.

local_system_simtimeÔÉÅ

This clock will drive the time in logical time steps in resolution of the property FEP3_CLOCK_SERVICE_CLOCK_SIM_TIME_STEP_SIZE. If for example the property is set to the value 5000000 ns the clock will set the time steps 0 ns, 5000000 ns, 10000000 ns and so on. The clock will try to set the time steps according to the clock of the operating system. So the difference between the step at time 0 and 5000000 should actually be 5000000 ns.

As the clock local_system_simtime is a discrete clock, for every new time step all Event Sink s will be called synchronously. If the processing of all Event Sink s takes longer than the FEP3_CLOCK_SERVICE_CLOCK_SIM_TIME_STEP_SIZE the clock will run slower than the wall clock. That would for example be the case if a Job with a cycle_time of 5000000 ns would take 8000000 ns to compute.

Also the speed of the time simulation can be adjusted using the factor FEP3_CLOCK_SERVICE_CLOCK_SIM_TIME_TIME_FACTOR. Setting it to FEP3_CLOCK_SIM_TIME_TIME_FACTOR_AFAP_VALUE will lead to a simulation running as fast as possible.

The clock‚Äôs real wall clock step size, which it tries to follow, is determined by dividing FEP3_CLOCK_SERVICE_CLOCK_SIM_TIME_STEP_SIZE by FEP3_CLOCK_SERVICE_CLOCK_SIM_TIME_TIME_FACTOR and may not fall below FEP3_CLOCK_WALL_CLOCK_MIN_STEP_SIZE.

local_system_realtimeÔÉÅ

The local_system_realtime clock uses the clock of the operating system. It will return the tick count of the current time starting at 0 (where usually 0 is the time stamp of starting the computer).
